## Notes

### Що будуємо

Mobile-first(!) AI-планер дня у стилі Todoist (https://www.todoist.com/). Юзер вивалює все, що в голові, голосом або текстом — AI перетворює хаос на структуровані задачі й план на сьогодні.

#### Головний флоу MVP який має працювати!

1. Capture: Поле «Що в голові?» текстом + мікрофон (опціонально). Пишеш/Диктуєш усе підряд.
2. Inbox: AI парсить інпути та переносить у задачі: пріоритет, час, дедлайн.
3. Today: Список задач на сьогодні сформований з беклогу

#### Наша ціль

1. Простий, але повноцінний продукт (MVP, який реально виконує свою функцію).
2. Cайт задоплеїний у світ (Та доступний за публічним посиланням).
3. Робимо 100% робочий продукт (Кожна заявлена функція працює. Поліруєш — скільки хочеш, але ядро має жити.)
4. Час на розробку (~3 дні), дедлайн буде 22 червня.

#### Рекомендовані фази реалізації

**Фаза 0 — розгорни інфраструктуру: спочатку живий URL**

Деплой — перший крок, а не останній — порожній сайт у світі цінніший за готовий продукт на локалхості.
Зроби деплой, поки ламатись нема чому.
Автоматизуйте і забудьте — один раз зʼєднуєш GitHub і Vercel, далі кожен push сам оновлює сайт.

**Фаза 1 — напиши продуктовий роадмеп: слова перед кодом**

Не пиши код, поки не сформулюєш словами, що будуєш — скіл brainstorming для цього й існує — він ловить питання, які інакше зʼїдять годину посеред кодування.
Краще якесь рішення, ніж ідеальне.
Спершу схема даних, потім код — єдине, що реально треба продумати до того, як писати код — це поля вашої «задачі». Це ваш головний контракт між інтерфейсом та AI.

**Фаза 2 — збудуй MVP один сценарій наскрізь**

Спочатку база, потім усе інше — хаотичний текст → структуровані задачі. Поки це не працює наскрізь, усе інше — декорації.

Секрети живуть на сервері, не в браузері — API-ключ — як пароль: усе, що потрапляє в код браузера чи git, стає публічним.

**Фаза 3 — допрацюй UX/UI очима користувача**

Тестуй на телефоні, бо продукт — для телефона — якщо не проходиш весь сценарій однією рукою в транспорті, mobile-first не вийшов.

Продумайте екран для нового користувача — коли людина тільки відкриває продукт, її список задач порожній. І це перше, що побачить журі на демо. Абсолютно порожній екран виглядає як баг.

**Фаза 4 — фінальні штрихи: спочатку надійно, потім вау**

Прод ≠ локалка: перевір на робочому URL — класична пастка — локально працює, на проді ні (через ключ,
якого немає на Vercel). Прогін роби тільки за реальним посиланням з телефона.

Диктування голос — це апгрейд, а не фундамент — спочатку текст має працювати бездоганно; голос додається поверх. Краще робочий текст, ніж зламаний мікрофон.

Бонус-функції — за співвідношенням «ефект до зусиль» — година до демо — не час починати Telegram-бота. Невелика корисна функція краща за велику недороблену.

#### Лайфхаки при розробці

- Спершу текст-роадмеп, потім код.
- Думай про цілісний продукт, а не набір фіч.
- Зроби якісний онбординг на старті.
- Спочатку робомо по-максимуму без бекенду та без постійного стореджу там де можливо, або на бекенді захардкоджені певні дані, веб аппка по можливості зберігає в local storage / session storage user-специфічні дані; але shared application секрети / паролі / API ключі НЕ мають бути просто захардкоджені ан фронтенді в браузері; спочатку без бази даних; login спочатку теж без справжньої аутентифікації, без збережння юзерів в базі даних; створення акаунту можливо просто зберігає креди юзера в local storage, (наприклад креди захардкоджені на бекенді); але фронтенд потрібно організувавати так, енкапсулювати так ці аспекти, щоб в майбутньому наступними кроками можно було реалізувати бекенд та базу даних з постійним сховищем замінивши реалізувацію інтерфесу наприклад!!!
- We must follow software development best practices: setup CI/CD, have unit / integration / e2e tests (tests pyramid), follow TDD, follow spec-driven development approaches; architecture must be thought upfront, be maintainable and extensible!!!
- follow software principles: SOLID Design Principles, SoC (Separation of Concerns), DRY (Don't Repeat Yourself), KISS (Keep It Simple, Stupid), YAGNI (You Aren't Gonna Need It) !!!


#### Рекомендовані приклади Advanced ideas

- Вибір AI-моделі (кілька, крім Gemini).
- Редагований шаблон промпту: «день фокусу» vs «адмін-
день».
- Telegram-бот: кидаєш задачі — падають в Inbox.

годин.

### 📊 Критерії оцінювання

1️⃣ Наскрізний сценарій — найважливіший критерій.

Ваш продукт має повністю закривати основний сценарій:

Brain dump → AI аналізує текст → створює задачі (пріоритет, час, дедлайн) → формує план на сьогодні.

Один завершений і стабільно працюючий сценарій отримає вищу оцінку, ніж багато незавершених функцій.

2️⃣ Продуктове мислення

Ми оцінюємо не кількість функцій, а якість ваших рішень.

Якщо ви вирішили додати нові можливості, вони мають логічно доповнювати продукт і робити його кориснішим для користувача.

Дві добре продумані функції можуть отримати вищу оцінку, ніж десять випадкових.

Для нас важливо побачити, що ви вмієте визначати, що дійсно варто будувати, а не просто додавати якомога більше можливостей.

3️⃣ Використання AI

AI має бути основою продукту, а не декоративним доповненням.

Ми звертатимемо увагу на те, наскільки AI допомагає користувачу виконати головний сценарій продукту.

4️⃣ Готовність продукту

Ваш MVP має бути готовим до використання.

Перевірте, що:

▫️продукт задеплоєний за публічним посиланням;
▫️відкривається без помилок;
▫️веб-версія працює на мобільних пристроях;
▫️усі заявлені функції працюють.

5️⃣ UX/UI

Ми також оцінюємо загальне враження від продукту.

Важливо, щоб ним було легко користуватися, а інтерфейс був зрозумілим і адаптованим під мобільні пристрої.


❗️Важливо:

Це не конкурс на найбільшу кількість функцій.

Ми оцінюємо не те, скільки ви встигли реалізувати, а наскільки добре ви зрозуміли задачу користувача та створили продукт, який її вирішує.

Core-функціонал є обов'язковим.
Після його реалізації можете розширити продукт власними ідеями. Ми будемо оцінювати не кількість додаткових можливостей, а те, наскільки вони логічно доповнюють продукт і створюють цінність для користувача.

💡 Не бійтеся виходити за межі базового функціоналу.
Якщо бачите можливість зробити продукт кращим — реалізуйте її.

Але пам'ятайте: найкращі додаткові функції — це не найскладніші, а ті, без яких продукт справді був би гіршим.
